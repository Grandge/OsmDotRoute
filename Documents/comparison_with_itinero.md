# OsmDotRoute と Itinero の比較・選定ガイド

[English](comparison_with_itinero.en.md) | 日本語

OsmDotRoute は [Itinero](https://github.com/itinero/routing) の単純な置き換えではなく、
**動的制約に特化した .NET 経路計算ライブラリ**である。
本書は「自分のユースケースに OsmDotRoute と Itinero のどちらが向くか」を判断するための、
できるだけフェアな比較を提供する。

> **想定読者**: .NET / OSM ルーティングの導入を検討している方、Itinero の既存ユーザー。
>
> **性能数値の出典**: OsmDotRoute 同梱ベンチ（`tests/OsmDotRoute.Benchmarks`）の実測値。
> 計測環境は Intel Core i7-1165G7 / Windows 11 / .NET 9。対象は愛知県津島市（市単位）および
> 愛知県・東京都（都道府県単位）の OSM データ。Itinero の数値は同一マシン・同一 RouterDb での実測。
> **推測値は用いない**。

---

## 1. 要約（TL;DR）

- **静的な経路計算 / 世界規模 / 実績重視 / Isochrone・マトリクス → Itinero**
- **実行中に通行制限が変化する / 大量のルート探査（マルチエージェント）/ 日本ローカル / System.* 完結 → OsmDotRoute**

OsmDotRoute の核心は「**シミュレーション実行中に進入不可・移動困難エリアを追加／削除し、
次の経路計算から再ビルドなしで即時反映する**」こと。
Itinero でもエッジのタグ付けとプロファイルの factor/speed 評価で同様のエリアを*表現*できるが、
その評価はエッジの静的属性から導かれ、プロファイル単位でキャッシュされる
（`ProfileFactorAndSpeedCache`）。そのため、実行中に対象エリアを頻繁に切り替える用途は
Itinero の設計の主眼ではない。OsmDotRoute はこの「実行時の即時 add/remove」をホットパスに据えている。
逆に、Itinero は CH（Contraction Hierarchies）や多対多計算など、OsmDotRoute が持たない
高速化・機能を備える。

---

## 2. 設計思想の違い

|              | Itinero                                                                | OsmDotRoute                                     |
| ------------ | ---------------------------------------------------------------------- | ----------------------------------------------- |
| 位置付け     | 汎用 OSM ルーティング                                                  | 動的制約特化のシミュレーション向けルーティング  |
| プロファイル | Lua / Native で広く拡張                                                | JSON 外部化（リビルド不要で調整）               |
| 制約の扱い   | タグ＋プロファイルで通行可否・速度を表現（実行時の頻繁な変更は不得手） | 実行時にエリアを add/remove、次回計算へ即時反映 |
| 適用範囲     | 世界中の OSM、多様なアルゴリズム                                       | 日本ローカル中心、Dijkstra に集中               |

Itinero は「世界中のあらゆる OSM ルーティング」に応える幅を持つ。
OsmDotRoute は幅を絞り、**動的制約のホットパス**に設計資源を集中している。

---

## 3. データ構造の違い

| 項目             | Itinero RouterDb (`.routerdb`)                                  | OsmDotRoute (`.odrg`)                                                              |
| ---------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| エッジモデル     | OSM タグを `edge_profile` に集約（共通 profile をエッジが参照） | エッジ単位でプロファイル評価結果を bake                                              |
| 空間インデックス | 頂点ベース                                                        | **エッジ STR R-tree（M=16）を bake**、動的制約の交差判定が O(log E)            |
| エッジ AABB      | 都度シェイプから計算                                              | **double × 4 を bake**、配列インデックスで O(1) 取得                          |
| エッジ属性       | profile / restriction 経由                                        | **14 bit フラグ**（橋・トンネル・高架・有料・私道・一方通行等を bake）         |
| エッジ形状       | コピーが発生しやすい                                              | **連続バッファ + `ReadOnlySpan<GeoCoordinate>`**（ゼロアロケーション）       |
| ファイルアクセス | ファイル全読み                                                    | **`MemoryMappedFile` + `ReadOnlySpan<T>`**（ゼロコピー、常駐メモリ最小化） |
| 入力             | OSM PBF / 中間 RouterDb 等                                        | **OSM PBF 直接抽出のみ**（`.odrg` 経由でランタイムへ）                       |
| ターン制限       | 対応                                                              | 未対応（Phase 4+ で形式予約のみ）                                                    |

OsmDotRoute の `.odrg` は「**動的制約を高速に評価するためのデータ土台**」として設計されている。
エッジ R-tree と AABB を事前 bake することで、ポリゴン／メッシュ制約とエッジの交差を
O(log E) で求め、結果を「制約 ID → 交差エッジ集合」としてキャッシュ。
Dijkstra のエッジ展開時は HashSet ルックアップ 1 回（O(1)）で済む。

## 4. アルゴリズムの違い

|      | Itinero                                                                  | OsmDotRoute（現時点）   |
| ---- | ------------------------------------------------------------------------ | ----------------------- |
| 探索 | Dijkstra / A\* / **CH** / 双方向 / 多対多 / Isochrone / マトリクス | **Dijkstra のみ** |

OsmDotRoute は現時点で Dijkstra のみ。A\* / 双方向 / CH は将来課題（Phase 4+）。

ここで重要なのは、**「動的制約 add/remove」と「CH 事前計算」は相性が悪い**こと。
CH は事前計算で高速化する手法だが、通行制限が変わるたびに再計算コストが発生するため、
ホットリロード型の用途では CH がかえって足かせになりうる。
OsmDotRoute は CH を持たない代わりに、後述のとおり制約 100 件下でも劣化させない設計を採る。

## 5. 性能特性の違い（津島市実測ベース）

> **比較上の重要な留保**:
> Itinero の数値は OsmDotRoute Phase 1 当時の実測値（同一マシン・**同一 RouterDb**・
> 43,685 頂点 / 57,331 エッジ）。Itinero 側は **CH を使わない通常探索**（`car.fastest`）での値であり、
> CH を有効化すれば Itinero はより高速になりうる。一方、OsmDotRoute の現行（`.odrg`）数値は
> 別データセット（53,727 頂点 / 74,276 エッジ、抽出範囲も広い）での実測のため、
> Itinero との Mean 直接比較ではなく**比率・桁オーダー**で読むこと。

### 5.1 経路計算（制約なし、津島市・100 ペア）

| 指標                      | Itinero（同一 RouterDb、CH なし） | OsmDotRoute（同一 RouterDb・Phase 1 当時） | OsmDotRoute（`.odrg` 現行） |
| ------------------------- | --------------------------------: | -----------------------------------------: | ----------------------------: |
| Mean                      |                          68.73 ms |      32.97 ms（Itinero 比**0.48x**） |                       7.70 ms |
| StdDev                    |                          20.98 ms |      2.80 ms（Itinero の**約 1/7**） |                       0.14 ms |
| 経路 1 本あたりアロケート |                          32.02 MB |                                   76.98 MB |             **3.12 MB** |

- **同一 RouterDb での Dijkstra 実装比較**では、OsmDotRoute は Itinero の **0.48 倍の時間**で完了。
  Lua インタプリタの実行揺らぎがない分、**StdDev は約 1/7** と安定している。
- **アロケート量は当初 Itinero より多かった**（77 MB vs 32 MB）が、`.odrg` + `ReadOnlySpan` 化で
  **3.12 MB まで削減**（Itinero 32 MB の約 1/10）。劣っていた点を設計刷新で逆転した。
- 経路同等性: 両方が経路を返した 89 ペアで距離乖離は平均 0.07%、全ペア ±10% 以内。
  さらに OsmDotRoute は Itinero が見つけられなかった 8 ペアでも経路を発見した
  （スナップ・探索の許容が広い）。

### 5.2 動的制約 100 件下（OsmDotRoute のみ）

|                              | Itinero                          | OsmDotRoute（`.odrg` 現行）                    |
| ---------------------------- | -------------------------------- | ------------------------------------------------ |
| 制約 100 件下の経路計算      | 同条件での直接比較は不可（後述） | **5.01 ms**（制約なし比 **0.65x**）  |
| 制約 add/remove スループット | —                               | **約 8,470 ops/sec**（1 サイクル 118 μs） |

OsmDotRoute は制約 100 件下でも**むしろ高速化（0.65x）**する。これは制約エリアで
Dijkstra の探索範囲が早期に枝刈りされ、かつ制約評価が O(1) ルックアップに圧縮されているため。

Itinero でもエッジのタグ付けとプロファイル設定で通行可否・速度は表現できるが、その評価は
エッジの静的属性に基づきプロファイル単位でキャッシュされるため、実行中に同じ 100 件を
add/remove して即時反映する標準的な手段はない。したがって本ベンチと同条件での直接比較は行えず、
ここでは OsmDotRoute 単独の値として示す。

### 5.3 ロードと規模

| 指標                   | Itinero                                        | OsmDotRoute（`.odrg`）                                           |
| ---------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| グラフのロード         | RouterDb デシリアライズ 115.4 ms（津島市規模） | MMF ゼロコピー。153–179 MB の都道府県データでも**0.2 秒台** |
| 都道府県単位の経路計算 | （未測定）                                     | 愛知県 117 ms / 東京都 288 ms（市単位は 8 ms 未満）                |

都道府県単位では Dijkstra のみだと 100 ms を超える（東京都 288 ms）。
大規模・高頻度クエリでは CH を持つ Itinero に分がある領域であり、OsmDotRoute は Phase 4+ で
CH 導入を検討する判断材料としている。

### 5.4 大量クエリ・マルチエージェント用途での含意

OsmDotRoute は、**マルチエージェントシミュレーションのように多数のエージェントが
各自の経路を頻繁に再計算する**、大量のルート探査が発生する場面での利用を主に想定している。
この場面では、1 クエリあたりの速度・揺らぎの小ささ・低アロケートが効いてくる
（津島市で 7.70 ms / StdDev 0.14 ms / 3.12 MB）。

Itinero の Lua プロファイルは、タグ評価ロジックをスクリプトで自由に記述できる**高い汎用性**が強みで、
世界各地の多様なルールに柔軟に対応できる。この柔軟性と引き換えに、スクリプト評価には
（factor の事前キャッシュ `ProfileFactorAndSpeedCache` で緩和されるとはいえ）
ネイティブ評価に比べ実行時のオーバーヘッドや揺らぎが生じやすい面がある。
Phase 1 の同一データ比較では、これが StdDev 約 7 倍の差（OsmDotRoute 2.80 ms 対 Itinero 20.98 ms）
として現れた。

OsmDotRoute はプロファイルの表現力を JSON ＋ ネイティブ評価に絞り、汎用性をある程度譲る代わりに
**大量クエリでの安定した速度**を優先している。どちらが適するかは、求める柔軟性と
スループットのバランス次第である。

## 6. 対応プロファイル

|          | Itinero                                    | OsmDotRoute                                                                    |
| -------- | ------------------------------------------ | ------------------------------------------------------------------------------ |
| 組込み   | Car / Pedestrian / Bicycle ほか多数（Lua） | Car / Pedestrian /**Bicycle** / **Truck（10t、日本道路法ベース）** |
| 拡張     | Lua / Native プラグイン                    | JSON 外部化（`VehicleProfile.LoadFromJsonFile`）                             |
| 地域適合 | 海外仕様ベースが中心                       | 日本国内を主眼（Truck は車両総重量 20t / 全高 3.8m / 全幅 2.5m を評価）        |

OsmDotRoute の Bicycle は通行可能エッジが全体の 99.5%（Car は 93.6%）で、
日本の自転車規則（高速道路・自動車専用道路以外は通行可）と整合している。

## 7. 依存とランタイム要件

|                | Itinero                            | OsmDotRoute                                                    |
| -------------- | ---------------------------------- | -------------------------------------------------------------- |
| ランタイム依存 | Itinero NuGet 群（複数アセンブリ） | **System.* 標準ライブラリのみ**（外部 NuGet ゼロ、net9.0） |
| 配布物         | 複数依存を伴う                     | 小さい。セキュリティ更新は .NET 本体追従で完結                 |

外部 NuGet が制限される環境・セキュリティ監査の負担を抑えたい環境では、
System.* 完結の OsmDotRoute が有利。

## 9. どちらを選ぶか — 用途別ガイド

### Itinero に向く用途

- 静的な OSM データに対する高速・高頻度の経路計算（CH が活きる大規模クエリ）
- 世界各地の OSM を統一プロファイルで扱う
- Isochrone / マトリクス / 多対多経路計算が必要
- 数年の運用実績を重視する
- Lua / Native でプロファイルを作り込む開発体制がある

### OsmDotRoute に向く用途

- **災害シミュレーション**（浸水・地震・道路閉鎖を動的に add/remove、再計算なしで反映）
- **マルチエージェントシミュレーション**（多数のエージェントが各自の経路を頻繁に再計算する大量クエリ。1 クエリの速度・安定性・低アロケートが効く）
- **自動運転 / 物流シミュレーション**（時間帯規制・事故閉鎖・工事区間を動的反映）
- **日本国内の Truck 配送計画**（10t 規模、`hgv` / `maxweight` / `maxheight` / `maxwidth` 評価）
- **実行中の制約変化が頻繁**な処理（add/remove スループット重視）
- **System.* 完結が必要な環境**（外部 NuGet 制限、監査負担軽減）
- **常駐メモリを抑えたい**（MMF ゼロコピー、複数 `.odrg` の使い分け）

### どちらでもよい / 慎重に評価すべき用途

- 動的制約なしの単純な 2 点間経路 → どちらも可。実績・機能網羅は Itinero、System.* 完結は OsmDotRoute
- 都道府県単位以上の超大規模・高頻度 → Itinero（CH）が有利。OsmDotRoute は Dijkstra のみで 100ms 超の領域
- 双方向 Dijkstra / CH / ターン制限が必須 → 現状 Itinero

---

## 10. OsmDotRoute が向かない用途

- **多対多マトリクス計算が中心** → Itinero / OSRM の方が機能が揃う
- **海外の OSM データを扱う** → Truck / Bicycle プロファイルは日本道路法ベースのため海外で不適合の可能性。
  Car / Pedestrian は汎用設計だが海外実績は未確認
- **CH 等を前提とする大規模・高頻度クエリ** → 現状は Dijkstra のみ
- **長期の本番稼働実績が必須** → OsmDotRoute は新興。Itinero には数年の実績がある

---

## 11. 将来計画（Phase 4+）

現時点で OsmDotRoute が劣る項目の多くは、将来の解消候補である。

- CH（Contraction Hierarchies）/ 双方向 Dijkstra など高速化アルゴリズム
- ターン制限（PBF Relation `type=restriction`）
- ユーザー定義プロファイルの抽出ツール対応
- NuGet 公開、マルチプラットフォーム（macOS / Linux）検証

---

## 12. 謝辞

OsmDotRoute は開発初期（Phase 1）に、グラフ構造を Itinero の `RouterDb.Network` から借用して
経路探索エンジンの動作確認を行った。その後 Phase 2 / Phase 3 で独自グラフ形式 `.odrg` へ移行し、
ランタイムから Itinero 依存を完全に撤去している。
優れた先行実装である Itinero と、その作者・コミュニティに敬意を表する
（参照のみを行い、ソースコードの複製は行っていない）。
