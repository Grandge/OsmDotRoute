## 1. このプロジェクトが生まれた経緯

本プロジェクトは、`d:/workspace/災害廃棄物処理シミュレーション/` プロジェクトのスピンオフとして発足した。

親プロジェクトでは Itinero 1.5.1 を OSM 経路計算ライブラリとして使っているが、以下の問題が浮上した：

1. **動的制約に対応していない**: 災害発生後の道路寸断や冠水を経路探索に反映できない
2. **Itinero がメンテナンス停止状態**: develop ブランチの最終コミットが 2023-09 で、約 2.5 年放置されている
3. **拡張の困難さ**: Itinero のソース調査の結果、`Profile.FactorAndSpeed()` や `WeightHandler.Calculate()` の引数にエッジ ID や座標が渡されない設計のため、空間制約を後付けする拡張ポイントが存在しないことが判明

これらを総合し、ユーザー判断として「Itinero の改造ではなく、独自の経路計算ライブラリを段階的に開発する」方針が確定した。本プロジェクトはその受け皿である。

詳細は親プロジェクトの以下のドキュメントを必ず確認すること：
- `d:/workspace/災害廃棄物処理シミュレーション/Documents/Itinero動的制約_調査結果.md`
- `d:/workspace/災害廃棄物処理シミュレーション/Documents/Itinero動的制約_PoC計画.md`

### Step B: 不明点の洗い出し

読了後、要件定義に進む前に**ユーザーに質問すべき項目**をリストアップする。最低限以下は確認する：

1. **プロジェクト名**: `OsmDotRoute` で確定（2026-05-18）
2. **ライセンス**: MIT / Apache 2.0 / プロプライエタリのどれか（オープンソース公開の有無）
3. **配布形態**: NuGet パッケージ / バイナリ / ソースのみ
4. **対応 .NET バージョン**: .NET 9 / 10 / それ以前との互換性
5. **ターゲット OS**: Windows / Linux / macOS / クロスプラットフォーム
6. **対応地域**: 日本のみ / グローバル
7. **メートル系のみ対応で OK か**（ヤード・マイル等の単位対応の要否）
8. **車両種別**: 車・歩行者の 2 種類で OK か、追加要件があるか
9. **動的制約の入力フォーマット**: ポリゴン（GeoJSON）/ バウンディングボックス / エッジ ID リスト等のどれを優先するか
10. **性能目標**: 1 経路あたりの応答時間目標（例: 100ms 以内）と、想定グラフ規模（市町村単位 / 都道府県単位 / 全国）
11. **公開時期目標**: いつまでに Phase 1 / Phase 2 / Phase 3 を完了させたいか
12. **開発体制**: ユーザー単独か、チーム参加者がいるか

### Step C: 要件定義書ドラフトの作成

ユーザーから上記の回答を得たら、`Documents/requirement_definition.md` をドラフトする。

含めるべき構成案：

```markdown
# OsmDotRoute 要件定義書

## 1. プロジェクト概要
   - 目的
   - 背景
   - スコープ（やること・やらないこと）

## 2. ターゲット利用シーン
   - 災害シミュレーション（最初の顧客）
   - 想定される第二・第三の顧客

## 3. 機能要件
   ### 3.1 必須機能（Phase 1 で実装）
   ### 3.2 将来機能（Phase 2-3）

## 4. 非機能要件
   - 性能
   - 対応プラットフォーム
   - 配布形態
   - 依存ライブラリ方針

## 5. インターフェース要件
   - パブリック API デザイン方針
   - 互換性方針

## 6. データフォーマット要件
   - 入力: OSM PBF（Phase 3 まで Itinero 依存）
   - 中間: 独自バイナリグラフ（Phase 2 で設計）
   - 出力: GeoJSON / Itinero.Route 互換

## 7. ライセンスと公開戦略
   - ライセンス
   - リポジトリ公開先
   - NuGet 公開の有無

## 8. 段階的開発計画
   - Phase 0/1/2/3 の詳細マイルストーン

## 9. 制約とリスク
   - 技術的リスク
   - スケジュールリスク

## 10. 用語集
```
